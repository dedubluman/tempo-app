## FUNCTIONALITY VERIFICATION

### Rate Limit Logic (Unchanged)
- 10 requests per 60 seconds per IP
- IP extracted from x-forwarded-for header
- Reset window: 60,000ms (60 seconds)

### Persistent Storage (NEW)
When KV_REST_API_URL and KV_REST_API_TOKEN configured:
- Rate limit data stored in Upstash Redis with 60s TTL
- Keys: `ratelimit:${ip}`
- Value: `{ count: number; resetAt: number }`
- Survives Vercel cold starts

### Fallback Mode (MAINTAINED)
When KV not configured:
- Falls back to in-memory Map (original behavior)
- Console warning emitted on module load
- Resets on cold start (same as before)

### Response Headers (NEW)
All responses include:
- X-RateLimit-Limit: "10"
- X-RateLimit-Remaining: "0" to "9"
- X-RateLimit-Reset: Unix timestamp (seconds)

429 responses also include:
- Retry-After: seconds until retry allowed

## GOTCHAS

### 1. @vercel/kv Deprecation
Initially attempted to use @vercel/kv but received deprecation warning:
```
@vercel/kv@3.0.0 deprecated. Vercel KV moved to Upstash Redis.
```
Solution: Use @upstash/redis directly.

### 2. Type Narrowing
TypeScript couldn't narrow union type in ternary:
```typescript
// ❌ Type error
'X-RateLimit-Remaining': String(rateLimit.limited ? 0 : rateLimit.remaining)
```
Solution: Extract value before creating headers object:
```typescript
// ✅ Works
const remaining = rateLimit.limited ? 0 : rateLimit.remaining;
const headers = { 'X-RateLimit-Remaining': String(remaining) };
```

### 3. Redis TTL
Redis TTL must be calculated dynamically when updating existing entry:
```typescript
const ttl = Math.max(1, Math.ceil((existing.resetAt - now) / 1000));
await kv.set(key, updated, { ex: ttl });
```
This ensures the key expires at the correct window reset time.

- KV_REST_API_URL and KV_REST_API_TOKEN must NOT have NEXT_PUBLIC_ prefix (server-only)
- Rate limit by IP can be bypassed with proxy rotation (known limitation)
- Consider adding user-based rate limiting in future (requires auth)
